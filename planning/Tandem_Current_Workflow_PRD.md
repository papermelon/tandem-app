# Product Requirements Document: Tandem Current Prototype

**Version:** 0.1  
**Status:** Implemented prototype baseline  
**Product Name:** Tandem

## 1. Summary
Tandem is a mobile-first, local-first caregiving coordination web app for a private care group called a **Tandom**. Each Tandom is identified by a **TanID** and is designed to help family caregivers and domestic workers share the same care record, complete in-person handovers, and track daily responsibilities with a single source of truth.

The current prototype is implemented as a Next.js Progressive Web App with IndexedDB persistence, seeded demo data, and an offline shell. It focuses on demonstrating the full interaction model rather than production integrations.

## 2. Goals
- Give caregivers one shared operating view of current instructions, tasks, notes, and handover context.
- Preserve **Ground Truth** as a versioned record, with explicit conflict resolution and traceable decision provenance.
- Support an in-person **Trust Bridge** handover flow using QR/token exchange.
- Provide a lightweight, in-app notification system for time-sensitive coordination.
- Allow users to query the record in plain language through a **data-only** assistant that cites stored records.

## 3. Current User Model
### 3.1 Tandom
A Tandom is the shared care space for one care recipient.

Current fields:
- `name`
- `tanId`
- `createdAt`
- `inviteToken`
- `memberIds`

### 3.2 Members
Each caregiver belongs to the Tandom and has:
- `displayName`
- `role`: `primary`, `family`, or `fdw`
- `language`: English, Tagalog, Bahasa, or Burmese

The prototype supports switching the active caregiver in-app to simulate multiple users.

## 4. Core Workflows
### 4.1 Join a Tandom
Purpose: add a caregiver to the private care group without public sign-up.

Current flow:
1. Tandem generates an invite token and QR code for the Tandom.
2. A new caregiver enters or pastes the token into the join form.
3. The caregiver provides name, role, and language.
4. Tandem adds the caregiver to the Tandom member list.

Current prototype behavior:
- Join is local and trust-based.
- There is no authentication or server validation yet.

### 4.2 Ground Truth Management
Purpose: keep the latest care instructions visible and versioned.

Current sections:
- Medication
- Mobility
- Mood
- Diet
- Appointments
- Emergency

Current flow:
1. The latest Ground Truth version is displayed as the current authoritative record.
2. A caregiver can submit a new chosen value for one section.
3. The caregiver must include a rationale.
4. Tandem creates:
   - a new **Decision Record**
   - a new **Ground Truth version**
5. The new version becomes the latest view.

### 4.3 Conflict Resolution with Decision Records
Purpose: make every override or conflict decision retrievable later.

Each Decision Record currently stores:
- conflict type
- affected field
- competing candidate values
- chosen value
- rationale
- evidence references
- impacted Ground Truth versions
- creator and timestamp

Current workflow behavior:
- Decision Records are immutable once created in the prototype flow.
- Ground Truth versions link back to `decisionRefs`.
- Handover receipts and query results can reference these decisions.

### 4.4 Trust Bridge Handover
Purpose: support an in-person caregiver handoff.

Current flow:
1. A caregiver starts a new handover.
2. Tandem generates a short-lived handover token and QR code.
3. The incoming caregiver pastes or scans the token value into the app.
4. Tandem marks the handover as claimed.
5. The caregiver completes a checklist:
   - Medication
   - Mobility
   - Mood
   - Where supplies are
6. Tandem generates a receipt after completion.

Receipt currently includes:
- reviewed sections
- summary notes
- linked decision references

### 4.5 Care Coordination
Purpose: manage daily work and narrative observations.

#### Tasks
Users can:
- create a task
- assign it to a caregiver
- set a due date/time
- cycle status between `open`, `claimed`, and `done`

#### Diary
Users can:
- add narrative care notes
- include lightweight hashtag-style tags
- review a timeline of recent entries

### 4.6 In-App Notifications
Purpose: surface important coordination events within the app.

Current notification types:
- task assigned
- task overdue
- handover started
- handover pending
- ground truth updated

Current behavior:
- Notifications are shown in a right-side notification center.
- Users can mark notifications as read.
- Notifications are generated by app logic and stored in local state.
- There is no OS-level push yet.

### 4.7 AI-Assisted Ingestion
Purpose: turn pasted text into reviewable care updates before changing Ground Truth.

Current draft types:
- photo
- voice

Current workflow:
1. User selects draft type.
2. User enters an asset label and pasted text.
3. Tandem heuristically proposes structured fields.
4. The user manually verifies each proposed field.
5. Once all fields are verified, the draft can be committed into Ground Truth.

Current prototype behavior:
- AI extraction is simulated with heuristic parsing.
- No live OCR, speech-to-text, or model API is wired in yet.
- Verified fields create a new Ground Truth version.

### 4.8 Ask Tandem
Purpose: answer caregiver questions using only stored records.

Current searchable sources:
- Ground Truth versions
- Decision Records
- Tasks
- Diary entries
- Handover receipts
- Ingestion drafts

Current workflow:
1. User enters a natural-language query.
2. Tandem searches local records.
3. Tandem returns:
   - an answer
   - citations to the supporting records
   - a fallback suggestion if nothing relevant is found

Current prototype behavior:
- Answers are **data-only**.
- No external medical advice is generated.
- Matching is rule-based and local.

## 5. Data Storage and Architecture
### 5.1 Frontend
- Next.js App Router
- React client-side state
- Mobile-first single-page workflow

### 5.2 Persistence
- IndexedDB stores the full application state
- The app seeds initial demo data on first load
- State persists locally across refreshes

### 5.3 Offline Support
- Web app manifest is present
- Service worker caches the app shell
- Prototype supports basic offline reopening after load

## 6. Current Seeded Demo Scenario
The prototype ships with a sample Tandom for **Ah Ma** with:
- three caregivers
- one medication conflict already resolved
- one Decision Record
- one completed handover
- active tasks
- a diary note
- one committed ingestion draft
- one notification

This seeded state is intended to make the workflows testable immediately on first launch.

## 7. Known Gaps Between Prototype and Vision
- No real authentication or secure invite validation
- No multi-device sync or backend
- No live QR camera scanner flow
- No real AI integrations for OCR, translation, or transcription
- No audit-grade permissions model
- No production reminder engine or OS push notifications
- No localization UI yet beyond stored caregiver language preference

## 8. Success Criteria for the Current Prototype
The current build is considered successful if a reviewer can:
1. Open the app and inspect a seeded Tandom.
2. Add a caregiver to the Tandom.
3. Create a Ground Truth update and see a linked Decision Record.
4. Start and complete a Trust Bridge handover.
5. Create and update tasks and diary notes.
6. Generate and clear in-app notifications.
7. Create an ingestion draft, verify it, and commit it.
8. Ask a question and receive a cited, data-only answer.

## 9. Recommended Next Steps
- Add real OpenAI and/or SEA-LION integrations for parsing, query quality, and translation.
- Replace token paste simulation with camera-based QR scanning.
- Add optional backend sync for multi-device demos.
- Add stronger validation, permissions, and acknowledgment flows for shared caregiving.
